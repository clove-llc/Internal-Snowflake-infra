# 07章 チェックリストの解答

各項目を「自分の言葉で説明できるか」の答え合わせ用。表現は違ってよく、要点が入っているかを見る。

## init / plan / apply / destroy を説明しながら実行できる

- `init`: provider の取得と作業ディレクトリの初期化。最初の1回と、provider や backend の構成を変えたときに実行する
- `plan`: `.tf`(あるべき姿)・state(把握している現状)・実環境(本当の現状)を突き合わせて実行計画を表示する。環境には触らない
- `apply`: 計画を実環境に適用する。環境が変わるのはここだけ
- `destroy`: state に載っている全リソースを削除する

## plan の `+` / `~` / `-` と `(known after apply)` を読める

- `+` 作成 / `~` 変更 / `-` 削除。`-/+` は「削除して作り直し」で、変更不可能な属性(名前変更など)を変えたときに出る。データを持つリソースでは要警戒
- `(known after apply)`: apply するまで確定しない値。他リソースの属性を参照している箇所に現れる

## リソース間参照を使って複数リソースを定義できる

`snowflake_database.tutorial.name` のように `リソースタイプ.コード内名.属性` で参照する。直書きと違い、値の変更が参照側へ自動で波及し、依存順序(DB → スキーマ)も参照から自動で決まる。

## variable / locals / output を使い分けられる

- `variable`: 外から値を受け取る口。tfvars や `TF_VAR_*` で渡す。環境差の吸収に使う
- `locals`: コード内だけで使う共通値。外から変える必要がないものに使う
- `output`: apply 結果から値を取り出す口。他システム(dbt、CI)への連携点になる

判断基準は「その値を誰が変えるか」。利用者が変えるなら variable、書き手の整理なら locals。

## state の役割と、リモートバックエンドが必要な理由

state は「Terraform が管理しているリソースの一覧と属性の記録」で、差分計算の基準。Terraform は state に載っているものしか管理しない。

ローカル state は (1) 共有できない (2) 失うと管理不能 (3) 同時実行で壊れる (4) シークレットが平文、の4点でチーム開発に耐えない。state をクラウドに置きロックを付けたリモートバックエンド(S3 / HCP Terraform 等)がその解決策。

## 「変更は必ず PR 経由」の理由

コードが唯一の正であり、手作業の変更は次の apply で上書きされて消えるため(ドリフト)。逆に管理外に作ったリソースは検出すらされず野良化する。変更を PR に一本化することで、レビュー・履歴・実環境の一貫性がすべて保たれる。
