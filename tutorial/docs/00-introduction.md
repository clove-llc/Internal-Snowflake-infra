# 00. IaC の動機と Terraform の位置づけ

コードは書かない章。ここの理解が以降すべての判断の土台になる。

## 手作業の環境構築が破綻する理由

Snowsight から GUI や SQL で環境を作ること自体は簡単だが、チームで続けると必ず同じ問題に行き着く。

- 環境の全体像を誰も説明できない(作った本人も忘れる)
- いつ・誰が・なぜ変更したかの記録がない
- 変更が実行されるまで、その妥当性を誰もレビューできない
- 「dev と同じものを prod に」が手作業のなぞり直しになり、差異が入り込む

IaC(Infrastructure as Code)はこれをコードの問題に変換するアプローチ。環境の構成をテキストで宣言し、実環境への反映はツールに任せる。コードになった時点で、Git の履歴が変更記録になり、Pull Request がレビューと承認のフローになり、同じコードから同じ環境を何度でも再現できる。

## Terraform の特徴は「宣言的」であること

シェルスクリプトは「DB を作れ、次にスキーマを作れ」という手順の記述で、2回実行すれば2回目はエラーになるし、途中で失敗した状態からの再実行も考慮が要る。Terraform は発想が逆で、書くのは手順ではなく「DB とスキーマが存在している」という**あるべき姿**だけ。現状との差分計算と、そこから必要な操作(作成・変更・削除)の導出は Terraform がやる。

この「実行前に差分(実行計画)を見る」という体験が Terraform の中核で、03章で実際に確かめる。

## Terraform と Snowflake の分担

Terraform 本体は特定サービスに依存しないエンジンで、各サービスの操作は provider というプラグインが担う。Snowflake 用は公式の [`snowflakedb/snowflake`](https://registry.terraform.io/providers/snowflakedb/snowflake/latest)。

このリポジトリで Terraform が管理するのは「環境という器」まで。

- 管理する: データベース、スキーマ、ウェアハウス、ロール、グラント、ユーザー
- 管理しない: テーブル内のデータ、日々の SQL、変換パイプライン(dbt などの領域)

## チュートリアルのゴール

- init / plan / apply / destroy を自分の言葉で説明しながら実行できる
- Snowflake の基本リソース一式(DB・WH・ロール・グラント)を HCL で定義できる
- state の役割と、リモートバックエンドが必要になる理由を説明できる
- このリポジトリの実環境資材に PR を出せる状態になる
